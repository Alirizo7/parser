# Самопроверка (selfcheck) — регрессионный гейт проекта

## Философия

Сравниваем сгенерированные документы с **эталонами клиента** (заполненные файлы
«…қилиниши керак» внутри архива, либо отдельные файлы у латинского клиента).
Сравнение — по СОДЕРЖАНИЮ (логическая карта «РМ → данные»), не построчно: позиции
строк в эталоне зависят от ручной разбивки по страницам.

Два сорта расхождений:
- **mismatch** — провал (ключевое поле не совпало);
- **note** — «к ручной проверке»: поля-эвристики и места, где эталон клиента сам
  расходится с его же исходными картами (наш вывод верен картам). На успех не влияют.

`CheckResult.ok == (total > 0 and not mismatches)` — пустое сравнение тоже провал.

## Эталонные числа (планка регрессии)

```bash
python manage.py selfcheck "Бухоро болалар стоматологияси.zip"
# 5_1б: совпало рабочих мест 41/41
# 6_5:  совпало ячеек 1032/1032   (= 7 реквизитов + 41 РМ × 25 колонок)
```

Ненулевой код выхода при провале — годится для CI. ⚠️ Фикстуры (`*.zip`, `*.doc`)
не в git — прогон воспроизводим только на машине с локальными файлами.

## Поиск эталонов в архиве — `find_reference(unpacked_dir, which)`

Рекурсивно по распакованному архиву; пропускаются файлы с «пустой/пустои» в имени
(пустые бланки). Критерии: `5_1b` — имя начинается с `5_1б` (кириллическая «б»!) и
содержит `керак`; `6_5` — начинается с `6 5` (с ПРОБЕЛОМ) и содержит `керак`.
Явные `--ref-*` пути имеют приоритет.

## Сравнение 5_1б — `compare_5_1b` (единица счёта: рабочее место)

`parse_5_1b` читает самую длинную таблицу → `{РМ: [(вещество_норм, процент), …]}`
(имя — lower без хвостовых точек; процент — первое целое). Правила:
- РМ есть в эталоне, нет у нас → mismatch (в bilingual — note: у RIZOYEV вещества
  дописаны клиентом вручную).
- Лишнее РМ у нас → всегда mismatch.
- Общие: точное равенство списков кортежей — **порядок веществ значим**.
- bilingual: имена веществ сравниваются после выброса скобок и fold.

## Сравнение 6_5 — `compare_6_5` (единица счёта: ячейка)

`_read_6_5`: реквизиты (7 полей; значение текстовых — жёстко колонка 2; коды —
regex по склеенной строке, `[БB]` — смешанные письменности) + данные РМ (строка с
`^\d{6}` в c0, все 26 grid-значений).

- Сравниваются 25 колонок — **c23 исключена** (дубль объединённой «Сут»:
  python-docx возвращает merged-ячейку дважды).
- c1 (должность): сравнение без дефисов/soft hyphen (U+00AD вшит в regex буквально!)
  /пробелов — автор эталона вставлял ручные переносы для узкой колонки
  (`ҳисоб-лаш`, `меҳ-нат`).
- **c22 «Сут» — расхождение уходит в notes**, не в mismatches: эталон Бухоро вручную
  ставит молоко=ҳа всем медикам (20 РМ) вопреки форме карты; мы выводим значение
  карты. Поэтому возможно `matched < total` при `ok == True`.

## Двуязычное сравнение — `compare_6_5_bilingual`

Наш шаблон — 26 grid-колонок, эталон RIZOYEV — 25: обе стороны сводятся к 25
логическим полям (`LOGICAL_6_5`); у нас выбрасывается c23. Равенство: должность —
fold без дефисов; код — только цифры; остальное — `fold(gen) == fold(ref)`
(`ҳа≡ha`, `йўқ≡yo'q` — различие письма не считается расхождением).

**`_REVIEW_6_5` = {4,5,6,7,8,9,10,11,12,18,22,24}`** — индексы `LOGICAL_6_5`,
расхождение по которым = note (био, аэроз, шум, инфра, ультра, обе вибрации, ЭМ,
ион, травмо, сут, **пенсия** — i=24 в логической схеме это пенсия). Ключевые
(расхождение = fail): т/р, должность, код, хим, микро, свет, тяж, напр, ОБЩ, ЯТҲВ,
отпуск, сокр, **лечпит** (i=23), реквизиты. ⚠️ Не путать логические индексы с
grid-колонками `_6_5_COL_NAMES` (там 24=лечпит, 25=пенсия) — `_REVIEW_6_5`
применяется именно к логическим.

## Команды

```bash
# Кириллический клиент (эталоны внутри zip):
python manage.py selfcheck "Бухоро болалар стоматологияси.zip" [--only 5_1b|6_5] [--workdir DIR]

# Латинский клиент (эталоны отдельными файлами, сверка между письмами):
python manage.py selfcheck KARTA.zip --bilingual \
    --ref-5-1b 5_1б_правилный.doc --ref-6-5 6_5-правилный.doc

# Извлечение в JSON без БД/UI (отладка якорей):
python manage.py extract_demo "<zip>" --out media/demo.json
```

Поведение selfcheck-команды: гоняет `run_pipeline`, рендерит документы во временную
папку, эталоны `.doc` конвертирует через LibreOffice. **Эталон не найден → WARNING
и пропуск (НЕ провал)**. Отчёт: `совпало {unit}: M/T`, до 40 mismatches (`✗`) и до
40 notes (`•`) — обрезка без пометки. Обе проверки выполняются, даже если первая
упала; провал любой → CommandError → exit 1.

`extract_demo` пишет `{"company_data", "workplaces", "warnings"}` (ensure_ascii=False)
и печатает предупреждения пайплайна.

## Сравнение Excel-протоколов — `compare_excel` (единица счёта: ячейка данных)

Сверяет сгенерированный xlsx с эталоном `excel_templates/N_*.xlsx` по СОДЕРЖАНИЮ
(РМ → под-строки). Единица — заполненная ячейка нормы/факта/времени/класса каждой
под-строки + итог блока. `read_excel_protocol` читает блоки по колонкам файла
(`_EXCEL_COLMAP`) и геометрии (`_EXCEL_GEOM`: фикс. число под-строк у файлов 2–5,
переменное у файла 1). Нормализация `_excel_norm`: число (запятая→точка), «йўқ»≡
«yo'q»≡«yo`q» (апострофы/пробелы выброшены), «-»/пусто → None (обе пустые не считаются).
У файла 1 сверяется и МЕТКА под-строки (кол. C = имя вещества, это данные); у файлов
2–5 метка статична (из прото) и в счёт не входит.

Классификация (как в docx — «верим картам», эталон Суд рукотворный):
- РМ есть в эталоне, нет у нас → **mismatch** (потеряли рабочее место);
- РМ есть у нас, нет в эталоне → **note** (эталон неполон: 000012a опущен в файлах 3/4;
  файл 5 в эталоне обрывается на 000040 — у нас 40 РМ с замером 1.4.x, эталон 33);
- расхождение ячейки на `_EXCEL_ANOMALY` = точечные пары (файл, РМ) {(1,000051),(2,000051),
  (4,000050),(4,000051)} → **note** (опечатки ввода эталона: 000051 — факт/время смещены,
  000050 — факт освещённости записан текстом). Маска ТОЧЕЧНАЯ: у 000051 в файлах 3/5
  данные верны, поэтому их регресс ПРОВАЛИТ гейт (не по всему РМ во всех файлах);
- прочее расхождение ячейки → **mismatch** (регресс извлечения/рендера).

**Два типа notes и их влияние на `matched/total`** (важно для чтения чисел):
- (а) *Сравнимая ячейка с известным расхождением* — обе стороны имеют ячейку, значения
  различаются, но расхождение отнесено к note (`_EXCEL_ANOMALY`): ячейка ВХОДИТ в
  `total`, но НЕ в `matched` → `matched < total`. Примеры: файл 1 **1572/1578**
  (6 ячеек 000051), файл 2 **263/265** (2 ячейки 000051), файл 4 **629/637**
  (000050/000051). Разница `total − matched` = число таких note-ячеек.
- (б) *Блок (РМ) есть только у одной стороны* — сравниваются лишь РМ из пересечения
  `set(ref) & set(gen)`, поэтому «лишний у нас» блок сравнимых ячеек НЕ даёт и НЕ
  меняет ни `total`, ни `matched`. Примеры: файл 3 **832/832** при note о лишнем
  000012a, файл 5 **165/165** при 7 notes о лишних админ-РМ. Счёт держится ровным,
  а факт «есть у нас, нет в эталоне» просто фиксируется строкой note.
- (потерянный блок наоборот — РМ эталона нет у нас — это **mismatch**, не note.)

Запуск: `python manage.py selfcheck <sud.zip> --excel-dir excel_templates`
(docx-проверки на архиве Суд сами пропускаются — там нет эталонов `…керак`).

**Планка регрессии Excel (Суд, lang=lat):** файл 1 — 1572 ячейки (6 notes 000051;
включает сверку имён веществ, кол. C), файл 2 — 263 (2 notes), файл 3 — 832
(1 note 000012a), файл 4 — 629 (9 notes), файл 5 — 165 (7 notes). Итого **3461**
совпавшая ячейка, 0 mismatch, ok=True.

⚠️ Уточнение к docx-планке: **6_5 даёт 1012/1032, НЕ 1032/1032** — 20 ячеек «Сут»
у медиков всегда уходят в notes (эталон Бухоро вручную ставит молоко=ҳа вопреки
картам). ok=True (notes не проваливают). 5_1б — 41/41.
