# Рендер: заполнение шаблонов 5_1б, 6_5, 6_4 (render.py)

## Общий приём

Берём фиксированный пустой шаблон-ассет из `attestation/assets/`, очищаем строки
тела ниже шапки и генерируем тело заново из датасета. Форматирование сохраняется
**клонированием строки-прототипа** на уровне lxml: `deepcopy(row._tr)` →
`table._tbl.append(tr)` (публичный API python-docx не умеет клонировать строку с
полным форматированием). Запись текста — `set_cell_text`: текст в первый run первого
параграфа (наследует форматирование), остальные runs опустошаются.

Ассеты: `template_5_1b.docx` (~2 МБ), `template_6_5.docx` (~1.7 МБ) — тяжёлые, из
клиентских файлов; `template_6_4.docx` (~14 КБ) — лёгкий чистый.

Все три функции: сортируют РМ по `workplace_sort_key`, в конце —
`_transliterate_doc(doc, lang)` и `doc.save`. При `lang="lat"` транслитерируется
КАЖДЫЙ run всего документа (тело + таблицы, включая вложенные) через `to_latin` —
и заголовки шаблона, и данные; поэтому весь остальной код работает только с
кириллицей. ⚠️ Колонтитулы и textbox-ы не транслитерируются.

## `_fill_reqs(table, company)` — реквизиты (общая для 6_5 и 6_4)

**Анти-утечка**: ВСЕГДА перезаписывает все ячейки значений; не извлеклось —
**очищает** (чтобы СТИР/ИФУТ/МХБТ компании-образца из шаблона не протекли другому
клиенту). Текстовые поля: якорь в cells[0] → значение в объединённую cells[2].
Коды: ячейка, чей `fold(text)` ТОЧНО равен метке из `_CODE_CELL_KEYS`
(`стир/ифут/мхбт` + смешанное `mxбt` — лат. m,x,t + кир. б) → значение в след. ячейку.

## render_5_1b(workplaces, out_path, *, lang)

Таблица `doc.tables[0]`; шапка = R0 (заголовки) + R1 («1|2|3»), прототип — R2.
Колонки: c0 — № РМ, c1 — вещества, c2 — `"NN %"` (пробел перед %).

- `group_substances`: группирует вещества по нормализованному проценту (порядок
  появления сохраняется); в ячейке первое вещество с заглавной, остальные со
  строчной: «Углерод оксиди, азот оксиди, силикат чанги (лой)».
- По строке на группу процентов; № РМ только в первой строке группы.
- **РМ без веществ** (флаг `substances_missing`) всё равно выводится одной строкой
  с пустыми c1/c2 — пропуск виден в документе, строки не теряются.

## render_6_5(company_data, workplaces, out_path, *, lang)

`doc.tables[0]` — реквизиты, `doc.tables[1]` — сводная. Grid 26 колонок
(25 логических: «Сут» объединена на 2 grid-колонки).

**Прототипы — ключевой трюк**: заголовок группы = R3 (одна ячейка на всю ширину);
строка данных = **R2 — строка нумерации «1|2|…|25», а НЕ пустая строка данных**.
Причина (задокументированный баг шаблона): у пустых строк данных объединение «Сут»
смещено на grid 21–22 вместо 22–23 — «сокр. день» и «Сут» схлопывались, когда
значения различались. Числа-плейсхолдеры перезаписываются данными. Удаляются
`rows[3:]` (R2 остаётся в шапке).

Группировка по подразделению — **по смежности** (`_group_by_subdivision`): новый
заголовок при смене `subdivision`; одноимённые группы через разрыв НЕ сливаются
(как в эталоне).

`row_values_6_5(rec)` — раскладка 26 значений:

| grid | значение |
|---|---|
| 0 | `workplace_no` |
| 1 | `position_from_perechen` или `position` (приоритет — «Перечень», как в эталонах) |
| 2 | `job_code` |
| 3–17 | факторы по `_FACTOR_COLS`: chem, biological, aerosols, noise, infrasound, ultrasound_air, vibration_general, vibration_local, em_field, ionizing, microclimate, lighting, severity, intensity, overall; пусто → `"-"` |
| 18 | `injury_risk` |
| 19 | `ppe_provided` |
| 20 | `benefits.extra_leave` |
| 21 | `benefits.reduced_hours` |
| 22, 23 | `benefits.milk` — ДВАЖДЫ (объединённая «Сут») |
| 24 | `benefits.therapeutic_food` |
| 25 | `privileged_pension` |

## render_6_4(company_data, workplaces, out_path, *, lang, warnings)

«Йиғма қайднома» — итоги по ПОДРАЗДЕЛЕНИЯМ. 13 grid-колонок:
c0 метка; c1 итого; c2–c5 классы условий труда 1–4; c6–c8 классы травмоопасности
1–3; c9 «Мос» / c10 «Мос эмас» / c11 «Ятҳв кўзда тутилмаган»; c12 «3–4 класс и/или
несоответствие» — **намеренно НЕ заполняется** (в эталоне пуст) → всегда `-`.

Каждый блок = 4 строки: заголовок + 3 строки данных (`_ROW_KINDS`: units /
employees / female — иш ўринлари / ходимлар / аёллар).

**Учитываются только позиции, сматченные со строкой «Перечня»** — фильтр по наличию
ключа `subdivision_6_4` (контракт с pipeline).

### Агрегация `_aggregate_group_6_4`

Вес строки: units — 1, но **«а»-суффиксные РМ весят 0** (доп. смена того же места,
сверено с эталонной суммой «Иш ўринлари, бирлик»); employees —
`_to_int(employees_count_6_4)` (число людей — из «Перечня», НЕ из карты); female —
`_to_int(female_count_6_4)`. Вес 0 → строка пропускается.

- Класс условий труда: старшая цифра `factors.overall` (`"3.2"→"3"`); не извлечён →
  warning (только на проходе units, чтобы не троить).
- Класс травмоопасности: `injury_risk_class_6_4`; не извлечён → warning.
- ЯТҲВ: `ppe_status_6_4 == "mos_emas"` → c10, ЛЮБОЕ другое (включая пустое) → c9
  «Мос»; НЕЗАВИСИМО: `ppe_not_envisaged_6_4` → c11. Колонки c9 и c11 намеренно
  пересекаются (эталон: Мос=114, кўзда=93).

Ноль рендерится как `"-"` (не «0») — `_fill_numeric_row`.

### Прототипная логика блоков подразделений (сердце функции)

Шаблон-ассет несёт шапку + блок «Корхона бўйича жами» (заголовок R3, данные R4–R6,
заполняются на месте по итогу ВСЕХ позиций) + готовые блоки подразделений с R10
шагом 4. На каждый рендер:

1. Прототипы нового блока (deepcopy заголовка и 3 строк данных первого блока)
   снимаются ДО любых удалений.
2. Каждый блок шаблона матчится по `fold(название) == fold(подразделение)` с первой
   ещё не использованной группой → заполняется НА МЕСТЕ (форматирование шаблона).
3. Подразделение «Перечня» без блока в шаблоне → **добавляется новым блоком в конец**
   + warning «…не найдено ни в одном блоке шаблона 6_4 — добавлен новый блок.»
4. Блоки шаблона, не сматченные ни с одним подразделением этого архива (заготовки
   чужого клиента) → **удаляются** (после добавлений, чтобы не сбить индексы).

Инвариант: набор блоков в готовом документе всегда равен набору подразделений
«Перечня» — независимо от того, под какое учреждение свёрстан шаблон-ассет.

Группировка 6_4 — **глобальная** по fold-ключу `subdivision_6_4` (в отличие от
смежностной в 6_5), порядок — первое появление, метка — оригинальный текст
разделительной строки «Перечня».

## Жёсткая геометрия шаблонов (сломается при смене вёрстки ассетов)

- 5_1б: шапка 2 строки, прототип R2.
- 6_5: шапка R0–R2, прототипы R2 (данные!) и R3 (группа).
- 6_4: итог компании R3–R6, блоки подразделений с R10 шагом 4.

## Мёртвый код в render.py

Импорт `mapping as M`, константа `_REQS_CODES`, функция `_norm` — не используются.

## Excel-протоколы лабораторных замеров (5 xlsx) — render_excel_1..5

Пять протоколов лаб. замеров по факторам условий труда, из ТОГО ЖЕ датасета.
Низкоуровневый слой — `services/xlsx.py` (openpyxl), сами рендеры — в `render.py`
(`render_excel_1`…`render_excel_5(company_data, workplaces, out_path, *, lang)`).

**Ассеты** (`assets/template_excel_1.xlsx`…`_5.xlsx`, строит `scripts/build_excel_assets.py`
из эталонов `excel_templates/`): полная шапка (строки 1..~25) + заголовки таблицы +
ОДИН прото-заголовок группы (файлы 1–4) + ОДИН прото-блок РМ. Данные остальных РМ
и лист `Лист1` (файлы 4/5) удалены. Прото-блок нужен ради форматирования (границы,
merge, формулы E/F/G).

**Клонирование** (`xlsx.emit_row`/`apply_merges`): строки тела генерируются
копированием стиля каждой ячейки прото-строки (`copy(cell._style)`), высоты и
объединений. Формулы псевдозамеров переносятся `openpyxl…Translator` — шаг (±0.1/
±1/±0.01) НЕ хардкодится, берётся из прото-формулы, ссылка правится на новую строку.
Порядок: сначала пишем ячейки (объединений ещё нет), потом создаём merge (запись в
merged-ячейку openpyxl запрещена).

**Раскладка колонок.** Файл 1 (12 колонок, СДВИНУТ): A№, B должность, C вещество,
D класс опасности (НЕТ в карте → пусто), E ПДК, F/G/H формулы `=I±шаг`, I факт,
J время, K класс, L итог по РМ. Файлы 2–5 (11 колонок): A№, B должность, C метка
под-строки (из прото), D норма, E/F/G формулы `=H±шаг`, H факт, I время, J класс,
K итог блока. № РМ/должность/итог — только 1-я строка блока (вертикальный merge).

**Источник данных** (карта → протокол):
- Файл 1: `substances` (norma/actual/pct/cls), порядок карты; итог L = `factors.chem`.
- Файл 2: `physical_measurements` (1.3.2 шум / 1.3.5 / 1.3.6 / 1.3.1); итог = max класс физ. факторов.
- Файл 3: `microclimate_measurements` — активная категория 1.8.x (факт непуст), метка
  `Ishlar toifasi` из 1.7.x; теплоизлучение 1.8.6 (факт «йўқ» → E/F/G/H = «йўқ»);
  итог = `factors.microclimate`.
- Файл 4: `lighting_measurements` (1.11.1/1.11.2/1.11.3, разряд «Vizual ish oqimi»);
  итог = `factors.lighting`. **Пульсация (1.11.6) и класс естеств. КЕО — НЕ выводятся**
  (эталон систематически их не сообщает; 0/52 РМ) — сознательно, чтобы совпасть с эталоном.
- Файл 5: `em_measurements` 1.4.1..1.4.10 → строки блока 0,1,2,3,4,6,8,10,12,14 (1.4.10 →
  строка 14, «VDT/PEHM»); итог = `factors.em_field`. **Включаются только РМ с замером 1.4.x.**

**Группировка**: файлы 1–4 — по подразделению (`_group_by_subdivision`, по смежности,
как 6_5); файл 5 — БЕЗ групп (все РМ подряд).

**Выборка РМ (какие РМ попадают в протокол):**
- Файлы 1–4 — **ВСЕ РМ** (`include=lambda w: True`). В частности «а»-суффикс `000012a`
  генерируется во всех пяти файлах. В эталонах Суд файлов 3/4 блока `000012a` НЕТ
  (клиент его там опустил), хотя карта `000012a` содержит и микроклимат, и свет
  (сверено) — поэтому у нас он выводится (по картам), а самопроверка отмечает его
  как **note** «есть у нас, нет в эталоне». Правило «генерим 12a везде» — сознательное
  («верим картам, не эталону»), симметрично тому, как 6_5/6_4 учитывают «а»-суффиксы.
- Файл 5 — **только РМ с замером ЭМИ**: `_has_em(w)` = есть непустой замер в ЛЮБОМ
  разделе 1.4.1–1.4.10 (у этого клиента заполнена только 1.4.10 — VDT/ПЭВМ). Даёт
  40 РМ; эталон Суд обрывается на `000040` и содержит 33. 7 «лишних» у нас —
  `000041–000046, 000048` (Bosh buxgalter, Buxgalter, Yuriskonsult, IT-специалист,
  инженер по ТБ, кадровик, делопроизводитель — админперсонал с ПЭВМ, у всех заполнен
  1.4.10); клиент их в эталон не внёс → **note** «есть у нас, нет в эталоне». Ни одно
  РМ эталона у нас НЕ потеряно (проверено). `000047` (без ПЭВМ) корректно не попадает.

**Двуязычие**: ассет держится на письме эталона (ЛАТИНИЦА). `xlsx.transliterate_region`
приводит управляемую часть (заголовки таблицы + тело) к `output_lang` «к целевому
письму»: `to_latin` — no-op для латиницы (метки эталона остаются точь-в-точь, без
round-trip `energiya→yenergiya`), `to_cyrillic` — для кириллицы. Статичная шапка
(приборы/НД) — «как есть», меняется только заказчик (`_fill_excel_reqs`: инлайн-строка
после метки «Наименование заказчика:/Buyurtmachi nomi:», анти-утечка как в docx).

⚠️ Отличие от docx-рендеров: те держат ассет на КИРИЛЛИЦЕ (эталон Бухоро), Excel —
на латинице (эталон Суд). Оба используют «транслитерацию к целевому письму», поэтому
работают для обоих клиентов; выбор письма ассета — из письма его эталона.
