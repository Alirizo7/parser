# Авто-аттестация — MVP

Веб-приложение, которое автоматически заполняет два фиксированных Word-шаблона
(`5_1б` и `6_5`) данными, извлечёнными из zip-архива аттестации рабочих мест.
Заменяет ручной копи-паст: **загрузил архив → проверил данные на экране → скачал
два готовых документа**.

Предметная область — аттестация рабочих мест (оценка условий труда) для организаций
Узбекистана. Документы на узбекской кириллице и русском.

---

## Возможности

- Загрузка zip-архива (drag-and-drop), фоновая обработка.
- Извлечение данных всех карт рабочих мест + реквизитов компании в **единый JSON**.
- Экран ревью с **инлайн-правкой** и подсветкой полей, требующих проверки.
- Генерация двух документов (`5_1б`, `6_5`) и скачивание.
- **Двуязычное извлечение** — карты на узбекской кириллице (Бухоро) и латинице
  (RIZOYEV) обрабатываются одним набором якорей (см. ниже).
- Самопроверка против эталонов: кириллица **5_1б — 41/41, 6_5 — 1032/1032**;
  латиница — совпадение по ключевым полям (см. раздел «Двуязычие»).

---

## Технологический стек

- Python 3.12, **Django 5.1**
- **LibreOffice (headless)** — конвертация `.doc → .docx` (старый `.doc` python-docx не читает)
- **python-docx** — чтение таблиц и заполнение шаблонов
- **Celery + Redis** — фоновая обработка (архив может быть ~100 МБ, десятки конвертаций)
- **SQLite** (код Postgres-ready — см. переменные `POSTGRES_*`)
- Фронт: Django templates + **Tailwind CSS (CDN)** + **HTMX** (поллинг прогресса,
  инлайн-сохранение) + **Alpine.js** (drag-and-drop)

## Быстрый старт (Docker)

```bash
docker compose up --build         # web + worker + redis; LibreOffice встроен в образ воркера
docker compose exec web python manage.py createsuperuser   # для админки /admin (опционально)
```

Откройте <http://localhost:8000> → «Новая загрузка» → перетащите zip → дождитесь
обработки → проверьте таблицу ревью → «Сформировать документы» → скачайте оба `.docx`.

- `web` применяет миграции и поднимает gunicorn; `worker` — Celery (LibreOffice внутри).
- Данные (загрузки, документы, БД) хранятся в docker-томах `media` и `db`.

## Локальный запуск без Docker

Нужен установленный LibreOffice (на macOS — `/Applications/LibreOffice.app`, путь
определяется автоматически; иначе укажите `SOFFICE_BIN`).

```bash
python -m venv .venv && source .venv/bin/activate
pip install -r requirements.txt
python manage.py migrate
python manage.py createsuperuser        # опционально
python manage.py runserver
```

При локальном запуске (`DEBUG=True`) фоновая обработка идёт в **отдельном потоке**
того же процесса (`ATTESTATION_TASK_RUNNER=thread`) — Redis и Celery не нужны.

### Полезные команды

```bash
# Прогон извлечения на примере → JSON (без БД и UI), для отладки якорей:
python manage.py extract_demo "Бухоро болалар стоматологияси.zip" --out media/demo.json

# Самопроверка: извлечь → заполнить → сверить с эталонами из архива.
# Падает с ненулевым кодом при расхождении (годится для CI):
python manage.py selfcheck "Бухоро болалар стоматологияси.zip"
python manage.py selfcheck "<zip>" --only 5_1b

# Латинский клиент: эталоны вне архива + сверка по содержанию между письмами
python manage.py selfcheck KARTA.zip --bilingual \
    --ref-5-1b 5_1б_правилный.doc --ref-6-5 6_5-правилный.doc
```

## Как устроен конвейер

```
zip → unpack → convert(.doc→.docx) → extract(карты + «Перечень») → единый JSON
                                                                       │
                                              ┌────────────────────────┴──────────────┐
                                       render 5_1б                             render 6_5
                                  (вредные вещества)                    (сводная, 25 колонок)
```

Главный приём: **не лить «карта → шаблон» напрямую**. Сначала всё извлекается в один
чистый промежуточный JSON (одна запись на рабочее место + блок реквизитов), затем оба
шаблона заполняются из него. Добавить третий шаблон = просто новый «рендерер».

Сервисный слой (в `attestation/services/`, **без зависимости от Django** — легко
тестировать и переносить в Celery):

| Модуль | Назначение |
|---|---|
| `unpack.py` | Распаковка zip (robust к кириллическим именам — в т.ч. UTF-8 без флага; фильтр мусора `~$`, `~WRL*.tmp`) |
| `convert.py` | Обёртка над LibreOffice (`.doc→.docx`); уникальный `UserInstallation` + блокировка (concurrency=1) |
| `normalize.py` | Нормализация значений: числа (`3. 3→3.3`), `ҳа/йўқ/yo'q/-`, классы, единицы |
| `mapping.py` | **Декларативный конфиг якорей** (правьте тут, не движок) |
| `extract.py` | Движок извлечения карт и «Перечня» → запись рабочего места |
| `render.py` | Заполнение шаблонов `5_1б` и `6_5` через python-docx |
| `selfcheck.py` | Поиск эталонов в архиве + логическое сравнение |
| `pipeline.py` | Оркестрация `unpack → convert → extract → датасет` |

Django-«клей»: `jobs.py` (загрузка `Batch`, обновление статуса), `tasks.py` (Celery),
`views.py` (экраны), модели `Batch` / `SourceFile`.

### Где якоря и как их менять

Все «якоря» (подстроки строк / номера разделов) — в
[`attestation/services/mapping.py`](attestation/services/mapping.py). Карты сгенерированы
из одного гос-шаблона, поэтому данные ищутся **по тексту**, а не по жёстким координатам
ячеек. Чтобы поправить маппинг под изменившийся шаблон — меняйте подстроки там, движок
трогать не нужно.

Примеры: подытоги факторов (`FACTOR_SUBTOTALS`), факторы по номеру раздела
(`FACTOR_SECTIONS`, напр. шум = `1.3.2`, ион. излучение = `1.12`), строки льгот
(`BENEFITS_ROWS`), порядок веществ для 5_1б (`SUBSTANCE_ORDER`).

## Двуязычие (кириллица + латиница)

Карты бывают на узбекской **кириллице** (Бухоро) и **латинице** (RIZOYEV).
Чтобы один набор якорей работал для обоих писем, и текст документа, и якоря
приводятся к общей форме через **транслитерацию** (`normalize.fold`): узб.
кириллица → латиница, апострофы (`oʻ/gʻ/ъ`) убираются, регистр сбрасывается.
Тогда `fold('баҳолаш') == fold('baholash')`, и якоря в `mapping.py` остаются
кириллическими. Подытоги заданы **наборами токенов** (а не цельной фразой), т.к.
грамматика шаблонов чуть различается (`Микроиқлим умумий` ↔ `Mikroiqlimni umumiy`).

Значения `ҳа/йўқ` извлекаются в обоих письмах (`ha`/`bor`/`yo'q` → канон).
Заполняется **один кириллический шаблон**. Самопроверка латинского клиента
(`--bilingual`) сравнивает значения, приводя обе стороны к одному письму
(`ҳа≡ha`, `йўқ≡yo'q`); различие языка ярлыков расхождением не считается.

### Язык выходных документов (селектор)

На загрузке и на экране ревью есть выбор **Кириллица | Lotin**
(`Batch.output_lang`, по умолчанию `cyr`). При рендере:
- `cyr` — ничего не транслитерируем (документ уже на кириллице);
- `lat` — после заполнения шаблона весь текст (`normalize.to_latin`) переводится
  в латиницу: и фиксированные заголовки, и данные. Цифры/коды/`%`/`-` не трогаются.

`normalize.py` содержит полные таблицы `to_latin()` (кир→лат, с `ye`-правилом и
апострофами `oʻ/gʻ`) и `to_cyrillic()` (лат→кир, диграфы длинным совпадением).

## Расхождения с исходным описанием задачи (верим файлам и эталону)

Подтверждено на реальных файлах; зафиксировано здесь намеренно:

1. **Номер рабочего места и должность** берутся ИЗ СОДЕРЖИМОГО документа — из
   абзаца `000002-сонли … ИШ ЖОЙИ НОМИ` (след. строка — должность), а НЕ из имени
   файла. Имя ненадёжно: бывает через подчёркивание (`3_Умумий_…doc`) или вовсе
   не совпадает (`Energetik.doc` при `Montajchi` внутри). **Папка карт** = та, где
   больше всего файлов-кандидатов (работает и `карта база`, и плоская `KARTA/`).
   Файл-карта без строки `…-сонли` → **громкое предупреждение** («Карты: найдено
   файлов N, распознано M, НЕ РАСПОЗНАНЫ: …»), а не молчаливая потеря.
2. **«а»-суффиксные РМ** (`000012а` — доп. место того же типа): сортируются сразу
   после базового номера по расширенному ключу (число, затем суффикс — любая
   буква кир/лат). Если в суффиксной карте пусты факторы / карты нет — данные
   копируются из базового номера (с предупреждением). Суффикс читается из
   содержимого (`000009а-сонли`).
3. **Код должности** берётся из документа **«Перечень»** (по номеру).
4. **Льготы (колонки 21–25) — ПРЯМО ИЗ КАРТЫ, раздел 4** (не эвристики):
   отпуск/сокр.день/питание/молоко — таблица 4.1 (колонка «Ҳақиқатда мавжудлиги»,
   `ha/bor→ҳа`, `yo'q→йўқ`); льготная пенсия — текст 4.2 («…пенсия таъминоти
   ҳуқуқи __ ҳа __»). ⚠️ Молоко: эталон **Бухоро** вручную ставит «ҳа» всем
   медикам (20 РМ) вопреки форме карты — мы выводим значение карты, а эти 20 ячеек
   в самопроверке Бухоро идут в «заметки» (не провал). У RIZOYEV/карты молоко из
   карты совпадает.
5. **Травмоопасность (c18)** не имеет детерминированного якоря — эвристика «медик/врач → 1,
   иначе → 2». Флаг `injury_risk_heuristic` (единственная оставшаяся эвристика).
6. **Должность** в 6_5 выводится чистым текстом — без ручных переносов-дефисов, которые
   автор эталона вставлял для узкой колонки (`ҳисоб-лаш`, `меҳ-нат`). Самопроверка сверяет
   должность без учёта дефисов/пробелов.
7. **Должность и код в 6_5 берутся из «Перечня»** (по номеру), как в эталонах обоих
   клиентов. Если имя должности в «Перечне» ≠ имени в карте на одном номере —
   флаг `position_mismatch` (подсветка «проверить вручную»), код молча не подменяется.
8. **Реквизиты компании всегда перезаписываются из карт; при неудаче ячейка очищается**
   (СТИР/ИФУТ/МХБТ компании-примера из шаблона не «протекают» к другому клиенту).
   Коды читаются и с пробелами (`311 452 380` → `311452380`).

### Латинский клиент (RIZOYEV) — известные расхождения с эталоном клиента

Эталоны RIZOYEV — **ручные** документы клиента; местами они расходятся с его же
исходными картами. Наш вывод **верен картам**; такие поля идут в «заметки»
самопроверки (не провал) и подсвечиваются в ревью:

- **5_1б, РМ 000001/000005/000007** — в картах этих мест **нет раздела веществ**
  (вещества в эталоне добавлены клиентом вручную). Рабочее место всё равно
  выводится в 5_1б с **пустой** ячейкой «Модданинг номи» и флагом
  `substances_missing` — пропуск виден в самом документе, строки не теряются.
- **Аэрозоли / вибрация / ЭМ-поле** — у нескольких РМ эталон клиента отличается
  от данных карты (напр. пыль из подраздела `1.1.6` клиент не отнёс к аэрозолям;
  значение локальной вибрации `1.3.5` проставлено как общая). Извлекаем по картам.
- **Названия веществ** клиент местами сокращает (`Silikat changi (loy)` →
  `Silikat changi`); единицы (`mg/m3`) срезаются, уточнения в скобках сохраняем.

Поля-эвристики (`milk_inferred`, `injury_risk_heuristic`, `pension_inferred`) и
указанные выше в ревью-таблице **подсвечены янтарным** с подсказкой «проверить вручную».

## Структура проекта

```
config/                 # Django-проект (settings, urls, celery, wsgi/asgi)
core/                   # общее приложение
attestation/
  models.py             # Batch (JSON-поля датасета), SourceFile
  jobs.py / tasks.py    # оркестрация + Celery-обёртки
  views.py / urls.py    # экраны и эндпоинты
  services/             # чистая логика конвейера (см. таблицу выше)
  assets/               # фиксированные пустые шаблоны (.docx): template_5_1b, template_6_5
  management/commands/  # extract_demo, selfcheck
templates/              # base + dashboard/upload/detail + HTMX-фрагменты
Dockerfile, docker-compose.yml
```

## Объём MVP

**Входит:** загрузка → фоновая обработка → извлечение → ревью с правкой и подсветкой →
генерация обоих документов → скачивание; самопроверка против эталона; docker-compose;
миграции; README.

**Не входит (на потом):** мультиарендность и роли сверх базовой админки; поддержка
произвольных новых шаблонов; OCR сканированных PDF; LLM-fallback включённым по умолчанию
(только опциональный хук за флагом `ATTESTATION_LLM_FALLBACK`, по умолчанию выключен).
